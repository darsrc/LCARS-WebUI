# LCARS UI Specification (v1.0)

**Project:** lcars-ui

**Package:** lcars_ui

**Schema Version:** 1.0.0

## 1. Overview

lcars-ui is a contract-first Python library designed to define LCARS (Library Computer Access and Retrieval System) interfaces programmatically. This project implements a **Server-Driven UI (SDUI)** architecture, specifically tailored for the strict aesthetic and functional requirements of Starfleet-style interfaces.

The core philosophy is the complete decoupling of the **Interface Definition** (Python/Backend) from the **Interface Rendering** (SPA/Frontend).

### 1.1 Architecture & Responsibilities

- Backend (The Source of Truth):
  - Written in Python using Pydantic models.
  - Defines the entire application state, layout, and logic.
  - Compiles the UI into a strict, versioned JSON Manifest.
  - Manages the event loop, processing incoming actions and dispatching state updates.
  - Constraint: The backend never generates HTML or CSS directly. It generates data describing the intent.
- The Contract (The Bridge):
  - Serves as the absolute boundary between backend and frontend.
  - Consists of two primary artifacts:
    - The Manifest: A static snapshot of the UI structure at load time.
    - The Protocol: A dynamic stream of events for runtime mutations.
  - Ensures type safety and version compatibility via strict schema validation.
- Frontend (The Renderer - Out of Scope):
  - A generic Single Page Application (React/Vue/Svelte) that acts purely as a consumption engine.
  - Fetches the Manifest to build the DOM tree.
  - Opens a WebSocket to listen for protocol events.
  - Constraint: The frontend contains no business logic. It only knows how to render a "Button" or a "Chart" based on the JSON description it receives.
## 2. Manifest Schema v1.0

The Manifest is the boot record for the application. It describes the static structure, initial state, and global configuration. It is generated by serializing the root Pydantic model at the endpoint GET /lcars/manifest.

**MIME Type:** application/json

### 2.1 Root Object

The root object defines the application shell and available routes.

```
{
  "meta": {
    "version": "1.0.0", // Schema version, strict semantic versioning
    "app_name": "Starship Operations", // Displayed in the browser title/main header
    "theme": "galaxy", // Controls the color palette (galaxy=TNG/DS9, nemesis=First Contact, tng=S1/S2)
    "lang": "en-US", // ISO language code for localization
    "sound_enabled": true // Hints to the frontend to load UI sound effects (beeps/chirps)
  },
  "layout": {
    // Global UI Shell Configuration
    "header": {
      "title": "USS Enterprise", // Top-left block title
      "subtitle": "NCC-1701-D", // Optional sub-registry text
      "color": "orange" // Base color for the header arch
    },
    "sidebar": {
      "position": "left", // Future-proofing: allow right-side or hidden sidebars
      "items": [
        // Navigation buttons always visible in the shell
        { "id": "nav_main", "label": "MAIN", "target_page": "main", "color": "blue" },
        { "id": "nav_eng", "label": "ENGINEERING", "target_page": "engineering", "color": "red" }
      ]
    }
  },
  "pages": {
    // Dictionary of Page Objects, keyed by page ID
    "main": { ... },
    "engineering": { ... }
  }
}
```

### 2.2 Page & Grid System

The layout system mimics the rigid, grid-based aesthetic of LCARS. Pages are strictly hierarchical: Page -> Rows -> Columns -> Widgets.

- Responsive Strategy: While specific pixel rendering is handled by the frontend, the schema provides hints (flex, width, height) to ensure the layout scales from PADDs (mobile) to Master Systems Display tables (desktop).
```
// Page Object
{
  "id": "main",
  "title": "Main Bridge", // Displayed in the specific page header area
  "rows": [
    {
      "id": "row_1",
      "height": "auto", // "auto" (content), "1fr" (flex), or "200px" (fixed)
      "columns": [
        {
          "id": "col_1",
          "width": "2fr", // Takes twice the space of col_2
          "widgets": [ ... ]
        },
        {
          "id": "col_2",
          "width": "1fr",
          "widgets": [ ... ]
        }
      ]
    }
  ]
}
```

### 2.3 Widget Nodes

Widgets are the atomic units of interaction. All widget definitions MUST adhere to the BaseWidget schema, ensuring consistent identification and state management.

**Common Fields:**

- id (Required, String): Unique identifier used for event targeting.
- type (Required, Enum): Discriminator field for the widget parser.
- label (Optional, String): Display text or accessible label.
- color (Optional, Enum): Standard LCARS colors: orange, red, blue, purple, white, yellow.
- disabled (Optional, Bool): If true, user interaction is blocked, and visual style is dimmed.
- visible (Optional, Bool): If false, the widget is removed from the DOM flow (not just hidden).
#### A. Primitive Widgets

Used for static display or simple status indication.

- Text: Renders typographic elements.
  - content: The text string.
  - size: h1 (Page Title), h2 (Section Header), body (Standard), mono (Data readouts).
- StatusTile: A classic LCARS block representing system health.
  - status: ok (steady color), warn (pulsing yellow), crit (flashing red).
  - value: Large numeric/text readout (e.g., "98%", "OFFLINE").
- Alert: A high-visibility banner.
  - severity: red (Red Alert), yellow (Yellow Alert).
  - message: The alert text.
  - blink: Boolean. If true, the alert element pulses opacity.
#### B. Input Widgets

Used to capture user intent. Interactions trigger upstream events.

- Button: A standard momentary action.
  - action_id: The event ID sent to the backend when clicked.
- Toggle: A persistent boolean state (ON/OFF).
  - checked: Initial boolean state.
  - action_id: Event ID sent on change.
- Select: A dropdown or cycle-button for choosing from a list.
  - options: List of { "label": "...", "value": "..." } objects.
  - value: The currently selected value.
  - action_id: Event ID sent on selection change.
- TextInput: A text entry field.
  - placeholder: Ghost text.
  - value: Current text.
  - password: Boolean. If true, masks characters.
  - regex: Optional validation string (frontend enforcement hint).
- Form: A logical grouping container for inputs. It does not render a visual border but aggregates data.
  - submit_label: Text for the submit button automatically generated at the bottom.
  - action_id: Event ID sent on submit.
  - children: List of nested Input widgets.
  - Note: When a form is submitted, the payload contains a dictionary of all child widget values.
#### C. Data Widgets

Used for visualizing structured data.

- Table: A strict row/column display.
  - headers: Array of strings.
  - rows: Array of row objects { "id": "...", "cells": ["...", "..."] }.
  - Update Strategy: Tables are updated by replacing the rows array via widget_update.
- LineChart / Sparkline: Time-series visualization.
  - series: Array of series objects { "name": "...", "data": [numbers], "color": "..." }.
  - x_labels: Array of strings matching the data length.
  - Sparkline Variant: Rendered without axes or gridlines, meant for embedding in small columns.
#### D. Media & Streaming Widgets

Complex widgets that handle their own data fetching or rendering loops.

- LogViewer: A scrolling terminal for system logs.
  - stream_id: A string ID linking this widget to a specific SSE/WS channel. The frontend automatically subscribes to log_chunk events matching this ID.
  - max_lines: Integer limit for client-side buffer (default 1000).
- VideoHls: An HLS (HTTP Live Streaming) player.
  - src: URL to the .m3u8 manifest.
  - autoplay: Boolean.
  - muted: Boolean.
  - Constraint: Does not support WebRTC. Uses standard HTML5 Video with HLS.js support.
- MicButton: A specialized push-to-talk controller.
  - upload_url: Endpoint to POST audio blobs to.
  - action_id: Event ID triggered after the backend processes the audio.
  - timeout_ms: Auto-stop recording after N milliseconds (default 5000).
## 3. Event Protocol v1.0

The runtime communication layer uses a strictly typed envelope protocol. The primary transport is **WebSocket**.

**Envelope Structure:**

```
{
  "v": "1.0",           // Protocol version. Mismatch causes client disconnect.
  "ts": 1715432000.123, // Unix timestamp (float) of event generation.
  "type": "string",     // discriminator
  "payload": { ... }    // flexible dictionary based on type
}
```

### 3.1 Server -> Client (Downstream)

These events mutate the UI state.

| Type | Payload Structure | Detailed Description |
| --- | --- | --- |
| manifest_update | { "path": "string", "value": any } | Patch Operation. Uses dot-notation (e.g., pages.main.rows[0]) to replace a subtree of the manifest. Used for major layout changes. |
| widget_update | { "id": "widget_id", "data": { "key": "val" } } | Delta Update. Targets a specific widget by ID and merges the data dictionary into the widget's properties. Example: Updating value on a StatusTile. |
| log_chunk | { "stream_id": "syslog", "lines": ["..."] } | Append Operation. Adds lines to a LogViewer. Optimized for high-frequency updates. |
| notification | `{ "message": "str", "level": "info" | "error" }` |
| action_ack | `{ "action_id": "btn_1", "status": "ok" | "fail" }` |

### 3.2 Client -> Server (Upstream)

These events represent user intent.

| Type | Payload Structure | Detailed Description |
| --- | --- | --- |
| action | { "id": "btn_1", "value": null } | Trigger. Standard button click or toggle switch. Value is usually null for buttons, or boolean for toggles. |
| input | { "id": "input_1", "value": "text" } | Data Entry. Sent on blur (focus lost) or debounced typing (if configured) for TextInputs. |
| form_submit | { "id": "form_1", "data": { "field": "val" } } | Batch Data. Sent when a Form container is submitted. data contains key-value pairs of all child inputs. |

### 3.3 Connection Lifecycle & Fallback

- Handshake: Client requests WS /lcars/ws.
- Validation: Server checks headers/auth (if implemented). Connection accepted.
- Sync: Server sends current state (optional, usually Client relies on Manifest fetched just prior).
- Loop: Bi-directional flow of events.
- Fallback (SSE): If WS fails/blocked, Client connects to GET /lcars/events (Server-Sent Events).
```
*Limitation:* SSE is one-way (Server->Client). Upstream actions must leverage POST /lcars/action/{id} HTTP endpoints.
```

  - Logs: LogViewer widgets work identically over SSE.
## 4. Audio/Mic Pipeline

The voice interface is designed for "Command & Control" (short phrases), not dictation.

**Workflow:**

- Interaction: User press-and-holds MicButton.
- Capture: Browser MediaRecorder API captures audio (WAV/WebM).
- Upload: On release, Client POSTs the Blob to /lcars/upload/audio.
- Processing (Server):
  - FastAPI endpoint receives multipart/form-data.
  - Invokes STTAdapter.transcribe(audio_bytes).
- Logic:
  - Transcribed text (e.g., "Red Alert") is passed to the internal Command Processor.
  - Command Processor determines UI changes.
- Response:
  - Server broadcasts widget_update (changing Alert widget to Red).
  - Server broadcasts notification ("Command Acknowledged").
**Pluggable STT Adapters:**

- MockSTTAdapter: Default for V1. Deterministic.
  - Input: Any audio file.
  - Output: Returns a string hash of the audio bytes or a fixed debug string. Crucial for reproducible CI/CD testing.
- CloudSTTAdapter: (Future/Plugin) Interfaces with OpenAI Whisper, Google Cloud Speech, etc.
## 5. API Endpoints

The API surface is minimal, frozen, and strictly typed.

| Method | Path | Request Body | Response | Description |
| --- | --- | --- | --- | --- |
| GET | /lcars/manifest | - | 200 OK (JSON) | Returns the full Manifest v1.0 object. |
| GET | /lcars/schema | - | 200 OK (JSON) | Returns the generated JSON Schema for the Manifest. |
| WS | /lcars/ws | - | Upgrade | Bi-directional event stream. |
| GET | /lcars/events | - | 200 OK (text/event-stream) | Fallback SSE stream for logs and updates. |
| POST | /lcars/action/{widget_id} | { "value": ... } | 200 OK or 202 Accepted | HTTP fallback for triggering widget actions. |
| POST | /lcars/upload/audio | Multipart (file) | 202 Accepted | Async audio processing. Returns immediately; results arrive via WS/SSE. |

## 6. Security & Versioning

### 6.1 Versioning Strategy

- Schema v1.0: The manifest structure is immutable for v1.x.
  - New widget types can be added in v1.x (backward compatible).
  - Removing fields or changing types requires v2.0.
- Protocol v1.0: The envelope structure is immutable.
  - Clients must check msg.v == "1.0". Mismatch triggers a hard reload or error screen.
### 6.2 Security Considerations

- CORS: By default, lcars-ui runs in "Development Mode" with allow_origins=["*"].
  - Production: Environment variable LCARS_CORS_ORIGINS must be set to restrict access to the specific frontend domain.
- Audio Permissions: Browsers strictly require a Secure Context (HTTPS) or localhost to access the Microphone API. The library cannot bypass this. Deployment behind an SSL-terminating proxy (Nginx/Traefik) is mandatory for audio features on non-localhost networks.
- Authentication: Explicitly out of scope for v1. The library assumes it is running within a trusted network (e.g., inside a VPN) or that the hosting FastAPI app implements its own middleware auth wrapper before the request reaches the lcars-ui router.
## 7. Plugin System (Backend)

The plugin system allows the core library to be extended without forking the codebase. It uses a dual-discovery mechanism.

### 7.1 Discovery Mechanisms

- Entry Points: The library scans the Python environment for packages declaring the lcars_ui.plugins entry point group.
- Filesystem: The library scans a local ./plugins directory for Python modules.
### 7.2 Plugin Capabilities

Plugins are restricted to specific extension points to maintain system stability:

- Page Registration: Plugins can inject new Page objects into the Manifest.
  - Conflict Resolution: Plugin pages are namespaced (e.g., plugin_name.page_id) to prevent overwriting core pages.
- Layout Injection: Plugins can request to add a generic navigation item to the Sidebar.
- Action Handlers: Plugins can register callback functions for specific action_id patterns.
  - Example: A plugin registers a handler for sensors_*. When a button sensors_scan is clicked, the plugin's function is invoked.
- Constraints:
  - Plugins cannot define new Widget Schema types in v1. They must compose UIs using existing primitives (Text, Table, Button, etc.).
  - Plugins cannot modify the Event Protocol envelope.
