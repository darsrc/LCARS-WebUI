# GitHub Publication Checklist

Only manual GitHub web UI actions belong here. Everything in this file is intentionally outside repo-local Codex scope.

## About section
- [ ] Set the repository description. Suggested starting point: `Python-first LCARS dashboard framework with a strict product renderer and a canonical target-bank acceptance harness.`
- [ ] Add topics such as `python`, `fastapi`, `react`, `typescript`, `lcars`, `dashboard`, and `server-driven-ui`.
- [ ] Verify the About-section links. If there is no external project site, consider linking to the latest release after it is published.
- [ ] Upload a social preview image if you want a custom GitHub card.

## Release draft / publish
- [ ] Create the release tag/draft that matches the intended publication version.
- [ ] Paste or adapt [RELEASE_NOTES.md](RELEASE_NOTES.md) into the release body.
- [ ] Keep the release body aligned with the closed Phase 18 baseline: `legacy_strict` is the product renderer, `phase14_family` is the canonical oracle / acceptance engine, `joern_strict` remains deprecated, and acceptance is limited to the closed 7-target / 4-family canonical bundle.
- [ ] Use autogenerated release notes under the curated release note if you want commit/PR detail.
- [ ] Carry forward the current security-audit caveat if the release text summarizes validation status; the caveat is documented in [RELEASE_NOTES.md](RELEASE_NOTES.md) and [lcars-ui/docs/RELEASE_READINESS_2026-03-23.md](lcars-ui/docs/RELEASE_READINESS_2026-03-23.md).
- [ ] Upload the local-only canonical acceptance bundle from `lcars-ui/artifacts/release_readiness_2026-03-23/canonical_acceptance/` as the release asset `LCARS-WebUI-v0.7.0-canonical_acceptance-2026-03-23.tar.gz`.
- [ ] Mark the release as Latest only if it corresponds to the closed Phase 18 / `lcars-ui 0.7.0` publication baseline.

## Repository settings
- [ ] Enable private vulnerability reporting if you want `SECURITY.md` to map directly to GitHub's security workflow.
- [ ] Verify default branch, visibility, and branch protection settings for the publication/handoff target.
- [ ] Enable GitHub Discussions if you want `SUPPORT.md` to route usage questions there instead of issues.
