## Why the target bank changes the strategy
- The current renderer work was shaped around specimen-specific parity learning. The target bank changes the acceptance question from “did one screen improve?” to “does the codebase reproduce a reusable LCARS grammar across multiple families and multiple states?”
- This makes page-based fidelity the wrong unit of progress. The unit of progress now has to be the target family.
- This also makes current explicit parity ID routing a temporary scaffold, not a long-term acceptance mechanism.
- The target bank forces a sharper separation between:
  - raw target evidence,
  - curated acceptance targets,
  - renderer primitives,
  - family composition rules,
  - deterministic renderer fixtures.
- If the migration is healthy, the number of reusable primitives and family recipes should grow faster than the number of frame-specific branches. If the opposite happens, the migration is failing.

## Target-bank organization model
- Keep `targets/` immutable. It is source evidence only.
- Add a target catalog layer. This should describe which target families exist, which frames are canonical, which are extended, which viewports are fixed, and what primitives each target is meant to exercise.
- Add a renderer-fixture layer. This should define deterministic inputs that reproduce canonical targets using code-rendered LCARS primitives.
- Organize target-bank concepts as:
  - `target family`: recurring LCARS grammar across one scene family.
  - `canonical target`: blocking frame used for acceptance.
  - `extended target`: non-blocking frame used to prove reuse or edge behavior after the canonical set passes.
  - `sequence probe`: a very small ordered subset of frames from one family used to confirm same-grammar/different-state behavior.
- Required catalog fields:
  - `family_id`
  - `target_id`
  - `source_path`
  - `viewport`
  - `tier`
  - `primitive_tags`
  - `stable_regions`
  - `dynamic_regions`
  - `acceptance_notes`
  - `sequence_group`
- Required organization rule:
  - A canonical target must be chosen because it forces a reusable primitive or family rule.
  - A canonical target must not be chosen merely because it is visually impressive.

## Migration phases
### Phase 1: Target-bank catalog and anti-cheat hardening
- Objective:
  - Convert the raw target bank into a controlled acceptance input and harden the repo against target-bank misuse.
- Preserved vs replaced boundaries:
  - Preserve raw `targets/` PNGs unchanged.

- Exact deliverables:
  - A canonical target catalog covering the 5 first-wave targets with family IDs, tiers, viewports, primitive tags, stable regions, and dynamic regions.
  - A target-bank usage doc that states raw targets are acceptance material only and never runtime-rendered.
  - New repo guardrails that fail if frontend/runtime code references `targets/` or imports target-bank assets.
  - A written preserved/transitional/remove-later inventory for current parity and strict-renderer subsystems.
- Likely files/systems touched:
  - `targets/` metadata layer added under repo docs or fixtures.
  - [`lcars-ui/docs/`](/home/darius/Documents/Projects/LCARS-WebUI/lcars-ui/docs)
  - [`lcars-ui/frontend/src/test/`](/home/darius/Documents/Projects/LCARS-WebUI/lcars-ui/frontend/src/test)
  - possibly [`lcars-ui/frontend/tests/visual/`](/home/darius/Documents/Projects/LCARS-WebUI/lcars-ui/frontend/tests/visual) only for catalog visibility, not renderer work.
- What is out of scope:
  - new renderer primitives,
  - target matching,
  - family implementation,
  - DSL changes,
  - visual diffing against target PNGs.
- Stop conditions:
  - the 5 canonical targets are fixed and documented,
  - each target has explicit acceptance metadata,
  - target-bank runtime misuse is covered by tests,
  - the repo has a written transitional/removal boundary for current parity code.
- Failure signals:

## Testing and acceptance strategy
- Blocking first-wave acceptance should require:
  - 5 canonical targets,
  - 3 target families,
  - at least 1 family passing in multiple states,
  - zero runtime references to `targets/`,
  - zero active page-ID or widget-ID coupling in the fidelity path.
- Required acceptance artifact per canonical target:
  - rendered output,
  - source target frame,
  - diff output,
  - viewport metadata,
  - target and family IDs,
  - pass/fail status.
- Existing structural tests should continue to verify:
  - strict composition regions exist,
  - unsupported markers do not appear on canonical targets,
  - width constraints and container ownership remain enforced.
- Anti-cheat tests must be expanded to enforce:
  - no runtime imports or asset URLs from `targets/`,
  - no `<img>`, SVG `<image>`, `drawImage`, `background-image`, `mask-image`, `image-set`, or `data:` raster usage in fidelity paths.
- Concrete proof that the migration is succeeding as a real LCARS system:
  - the same family recipe renders multiple accepted states,
  - two or more families share actual renderer primitives,
  - dense repeated LCARS grammar is supported in addition to sweep/elbow grammar,
  - canonical acceptance no longer depends on `overview`/`systems` naming.
- Concrete failure signals that indicate drift:
  - target/frame-specific branches grow faster than shared primitive code,
  - acceptance tests hardcode target names instead of family semantics,
  - comparison tolerance grows while renderer structure does not improve,
  - old parity-ID routing remains the easiest way to make tests pass,
  - the target catalog is bypassed during renderer work.

## Risks and anti-cheat implications
- Highest risk: target-bank leakage into runtime rendering.
  - Required response: explicit repo-wide tests that ban runtime references to `targets/`.
- High risk: old specimen coupling being renamed rather than removed.
  - Required response: phase-level success criteria must ban new frame-ID/page-ID fidelity branches.
- High risk: acceptance drift caused by keeping old goldens and old tests around too long.
  - Required response: Phase 6 must explicitly demote or remove them as authoritative signals.
- Medium risk: claiming system progress after one family.
  - Required response: no LCARS-ready claim until Seismographic, Holodeck, and Periodic Table are all in canonical coverage.
- Medium risk: widening scope into “all frames now.”
  - Required response: first-wave scope remains 5 canonical targets only.
